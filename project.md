## 4. МЕД-ОТДЕЛ v2 — СТУДИЙНАЯ СИСТЕМА САМООБУЧЕНИЯ

Главная фича. МЕД-ОТДЕЛ наблюдает за всей студией через единую шину событий.

### Архитектура — Шина событий

Все результаты задач пишутся в `memory/events_bus.json`. **Агенты сами не пишут в шину** — только кнопка "Оценить" в UI вызывает `POST /api/med-otdel/evaluate`, которая:
1. Берёт последний ответ агента из `chat_history`
2. Формирует событие с уникальным `task_id`
3. Атомарно пишет в `events_bus.json` (через `os.replace()`)
4. Запускает Critic

### Структура события

```json
{
  "id": "evt_001",
  "task_id": "task_abc123",
  "timestamp": "2026-04-03T02:00:00",
  "agent_id": "writer",
  "event_type": "task_completed",
  "result": "Сгенерированный промпт...",
  "status": "success"
}
```

### Три режима работы МЕД-ОТДЕЛА

| Режим | Триггер | Действие |
|-------|---------|----------|
| **agent_heal** | 2+ провала у одного агента подряд | Эволюция промпта агента (v1→v2→v3) |
| **chain_heal** | Провалы на стыке двух агентов (одинаковый `task_id`) | Эволюция промпта принимающего агента — добавляется правило обработки входных данных от конкретного отправителя |
| **studio_alert** | 50%+ агентов в статусе `error` | Пауза студии + уведомление пользователя в UI |

### Flow работы

```
Пользователь нажимает "Оценить" в панели агента
         ↓
POST /api/med-otdel/evaluate:
- Берёт последний ответ из chat_history
- Формирует событие с task_id
- Атомарно пишет в events_bus.json
- Запускает Critic
         ↓
Critic оценивает результат
         ↓
    ┌──────────────┐
    │   PASS?      │
    └─────────────┘
      YES  │  NO
           ↓
      Записать FAIL в events_bus.json
      Обновить статус агента → error
           ↓
      Fixer исправляет (по кнопке)
           ↓
      Critic оценивает снова
           ↓
      ┌──────────────┐
      │  PASS?       │
      └─────────────┘
        YES  │  NO
             ↓
      МЕД-ОТДЕЛ проверяет 3 режима:

      1. agent_heal: 2+ fail у одного агента подряд?
         → evolve_agent() → новый промпт v2

      2. chain_heal: fail на стыке A→B (тот же task_id)?
         → обновить промпт принимающего агента

      3. studio_alert: 50%+ агентов в error?
         → пауза + уведомление в UI

      ↓
      Лог в med_log.json + UI обновляется
```

### Статусы агентов

| Статус | Значение |
|--------|----------|
| `idle` | Простаивает, готов к задаче |
| `working` | Выполняет задачу |
| `waiting` | Ждёт ответа/результата |
| `error` | Провал оценки — МЕД-ОТДЕЛ лечит |

### Версионирование промптов

```
med_otdel/versions/writer_v1.json  ← архив
med_otdel/versions/writer_v2.json  ← архив
med_otdel/versions/writer_v3.json  ← ТЕКУЩИЙ (нет отката)
```

**Правило:** Откат запрещён. Только вперёд.

### Новые файлы МЕД-ОТДЕЛА

```

├── med_core.py          # Оркестратор 3 режимов
├── agent_memory.py      # Память агентов (порог эволюции = 2)

## 5. HR-АГЕНТ — НАЙМ ПОД ЗАДАЧУ

Когда задача не решается существующими агентами:

```
HR получает описание задачи
        ↓
Анализирует: какой агент нужен
        ↓
Генерирует: имя, роль, промпт, модель
        ↓
Critic проверяет нового агента
        ↓
Агент появляется в интерфейсе офиса
        ↓
Работает пока задача не выполнена
```

Временные агенты помечаются тегом `temp` и могут быть удалены после выполнения задачи.

## 14. РАБОЧИЙ ЦИКЛ OPENCODE

Обязательный цикл для каждой задачи. Никогда не пропускается.

### Этапы цикла

1. **Plan** — Строю план реализации задачи
2. **Critic** — автоматически оценивает план: находит ошибки, риски, противоречия
3. **Fixer** — автоматически исправляет замечания Critic в плане
4. **Critic** — автоматически перепроверяет исправленный план
5. **Повтор максимум 3 круга** Critic/Fixer
6. **После 3 кругов** — стоп, спрашиваешь пользователя
7. **"ок" от пользователя** → Build (реализация)

### Правила

- **Build только после чистой оценки Critic** — если Critic нашёл замечания, Fixer должен их исправить и Critic должен подтвердить
- **Critic/Fixer цикл никогда не пропускается** — даже для мелких задач
- **Пользователь видит только финальный план** + сколько кругов Critic/Fixer прошло
- **Максимум 3 круга** — если после 3 кругов Critic всё ещё находит проблемы, останавливаюсь и спрашиваю пользователя

### Формат вывода для пользователя

```
План: [краткое описание]
Кругов Critic/Fixer: N/3
Статус: [Готов к Build / Есть проблемы]
```

### Когда цикл запускается

- При получении новой задачи от пользователя
- При изменении требований в процессе
- При обнаружении критической проблемы во время Build
- Перед каждой новой фазой разработки
